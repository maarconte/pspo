## 2024-05-24 - Leaking Authentication Errors
**Vulnerability:** Raw Firebase `error.code` and `error.message` strings were concatenated and rendered directly to the user in the `AuthContainer` component upon failed login, registration, or OAuth attempts.
**Learning:** Returning unhandled upstream error structures back to the client leaks details of internal system libraries (e.g. `firebase/auth`) and creates a user enumeration risk where attackers can verify whether specific email addresses correspond to existing accounts (e.g., distinguishing between `auth/user-not-found` and `auth/wrong-password`).
**Prevention:** Always catch and handle errors on the client side, then map specific codes to generic error messages for end-users (e.g. "Identifiants invalides.") using centralized error handlers like `src/features/auth/utils/authErrors.ts`.
## 2024-05-24 - Leaking Authentication Errors in API Services
**Vulnerability:** Raw Firebase `error.message` strings were being re-thrown as Error objects in the API services layer (`authService.ts` and `roleService.ts`), eventually bubbling up to components where they could be displayed in toast notifications.
**Learning:** Returning unhandled or un-sanitized upstream error strings back to the client leaks details of internal system libraries and custom backend cloud functions. This provides a vector for account enumeration and leaking information regarding server-side implementations.
**Prevention:** Always strip raw errors at the API layer. Use centralized error mappers (like `getAuthErrorMessage`) for mapping standard error codes, and explicitly define safe generic fallback messages like "Authentication failed" or "Failed to assign role. Please ensure you have the required permissions." before throwing errors back to UI components.
