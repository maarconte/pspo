## 2024-05-24 - Leaking Authentication Errors
**Vulnerability:** Raw Firebase `error.code` and `error.message` strings were concatenated and rendered directly to the user in the `AuthContainer` component upon failed login, registration, or OAuth attempts.
**Learning:** Returning unhandled upstream error structures back to the client leaks details of internal system libraries (e.g. `firebase/auth`) and creates a user enumeration risk where attackers can verify whether specific email addresses correspond to existing accounts (e.g., distinguishing between `auth/user-not-found` and `auth/wrong-password`).
**Prevention:** Always catch and handle errors on the client side, then map specific codes to generic error messages for end-users (e.g. "Identifiants invalides.") using centralized error handlers like `src/features/auth/utils/authErrors.ts`.## 2026-04-16 - Prevent Leaking Stack Traces and Internal Errors
**Vulnerability:** The application was exposing raw backend error properties (e.g., `error.message` from Firebase Authentication or Cloud Functions) directly to the client UI, which can lead to information disclosure and potential system enumeration.
**Learning:** Passing raw `error.message` properties up through the frontend can accidentally expose internal system details, sensitive backend configuration, or allow attackers to enumerate users by analyzing slight differences in error messages.
**Prevention:** Catch external dependency errors at the API layer and return standardized, generic error messages (e.g., "Authentication failed" or "Failed to delete user") to the frontend.
