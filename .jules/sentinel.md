## 2024-05-24 - Leaking Authentication Errors
**Vulnerability:** Raw Firebase `error.code` and `error.message` strings were concatenated and rendered directly to the user in the `AuthContainer` component upon failed login, registration, or OAuth attempts.
**Learning:** Returning unhandled upstream error structures back to the client leaks details of internal system libraries (e.g. `firebase/auth`) and creates a user enumeration risk where attackers can verify whether specific email addresses correspond to existing accounts (e.g., distinguishing between `auth/user-not-found` and `auth/wrong-password`).
**Prevention:** Always catch and handle errors on the client side, then map specific codes to generic error messages for end-users (e.g. "Identifiants invalides.") using centralized error handlers like `src/features/auth/utils/authErrors.ts`.
## 2024-10-24 - Prevent Raw Third-Party API Error Message Leakage
**Vulnerability:** The application was exposing raw error messages directly from third-party APIs (like Firebase Auth, Firestore, and Cloud Functions) to the user interface via `error.message`.
**Learning:** This practice can inadvertently leak sensitive information about the application's internal structure, database schema, or third-party service configurations to potential attackers, aiding them in reconnaissance.
**Prevention:** Always catch raw third-party API error messages. Sanitize them by mapping known error codes to generic, user-friendly messages (e.g., using a utility like `getAuthErrorMessage`), or provide a safe, generic fallback message if the error code is unknown. Never display `error.message` directly in the UI or throw it where it will be displayed.
